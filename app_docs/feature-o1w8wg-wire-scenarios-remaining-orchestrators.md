# Wire Scenario Phases into Remaining Orchestrators

**ADW ID:** o1w8wg-wire-scenarios-into
**Date:** 2026-04-08
**Specification:** specs/issue-400-adw-o1w8wg-wire-scenarios-into-sdlc_planner-wire-scenarios-remaining-orchestrators.md

## Overview

Extends the `scenarioTestPhase` + `scenarioFixPhase` wiring introduced in #399 for `adwSdlc.tsx` to the four remaining orchestrators that have a test phase: `adwPlanBuildTest`, `adwPlanBuildTestReview`, `adwChore`, and `adwPrReview`. After this change, all five orchestrators follow the same unified pattern: `unitTestPhase` → `scenarioTestPhase` → orchestrator-level retry loop calling `scenarioFixPhase` on blocker failures. BDD scenario validation is now consistent across every workflow type.

## What Was Built

- `adwPlanBuildTest.tsx` wired with `executeStepDefPhase` + `executeUnitTestPhase` + scenario test/fix retry loop before PR phase
- `adwPlanBuildTestReview.tsx` wired with same pattern, plus empty-`scenariosMd` patch passed to review so review reads `scenario_proof.md` instead of running scenarios
- `adwChore.tsx` wired with same pattern; scenario testing runs before `executeDiffEvaluationPhase` so scenario-fix commits are visible to the diff evaluator gate
- `adwPrReview.tsx` replaces the former `executePRReviewTestPhase` with `executeUnitTestPhase` + scenario test/fix retry loop; all phases use `config.base` via `runPhase` (closure-wrapper pattern from #398)
- `scenarioRetries` counter added to state metadata for all four orchestrators
- BDD acceptance scenarios in `features/wire_scenarios_remaining_orchestrators.feature` covering all four orchestrators and cross-cutting retry loop behaviour

## Technical Implementation

### Files Modified

- `adws/adwPlanBuildTest.tsx`: replaced `executeTestPhase` with `executeStepDefPhase` + `executeUnitTestPhase` + scenario retry loop; added `scenarioRetries` to metadata
- `adws/adwPlanBuildTestReview.tsx`: same replacement; added `executeReviewWithoutScenarios` wrapper to patch `scenariosMd` to empty before review
- `adws/adwChore.tsx`: same replacement; scenario loop runs before `executeDiffEvaluationPhase`, diff gate and conditional escalation unchanged
- `adws/adwPrReview.tsx`: removed `executePRReviewTestPhase` import and call; replaced with `executeInstallPhase` + `executeUnitTestPhase` + scenario retry loop all via `runPhase(config.base, ...)` using `CostTracker`
- `features/wire_scenarios_remaining_orchestrators.feature`: BDD feature file with `@adw-400` and `@regression` scenarios for the new wiring
- `features/step_definitions/wireStepdefPhaseIntoOrchestratorsSteps.ts`: step definitions supporting the new feature scenarios

### Key Changes

- **Unified retry loop pattern** — identical `for (attempt = 0; attempt < MAX_TEST_RETRY_ATTEMPTS; attempt++)` structure in all five orchestrators; exits on first passing result, does not hard-fail on exhaustion
- **Empty `scenariosMd` patch for review** — `adwPlanBuildTestReview` creates a `patchedConfig` via shallow clone before calling `executeReviewPhase`, matching the `adwSdlc` pattern; review reads the pre-written `scenario_proof.md` rather than re-running scenarios
- **`config.base` for PR review** — `adwPrReview` uses the `PRReviewWorkflowConfig.base` (`WorkflowConfig`) when calling `executeUnitTestPhase`, `executeScenarioTestPhase`, and `executeScenarioFixPhase` via `runPhase`, consistent with the closure-wrapper pattern from #398
- **Diff evaluator ordering in `adwChore`** — scenario test/fix loop runs before `executeDiffEvaluationPhase` so any scenario-fix commits are included in the evaluated diff
- **`executePRReviewTestPhase` retained but unused** — the function remains in `prReviewCompletion.ts` as an export for backward compatibility; it is no longer called by `adwPrReview.tsx`

## How to Use

The wiring is transparent to operators — orchestrators automatically run scenario testing after unit tests. No configuration changes are required unless you want to enable or configure scenarios for a target repository:

1. Ensure BDD feature files exist in the target repo (or are generated by the scenario phase)
2. Set `## Run Scenarios by Tag` in `.adw/commands.md` to a valid Cucumber tag (e.g. `@regression`); if absent or set to `N/A`, `scenarioTestPhase` skips silently
3. Optionally configure `## Start Dev Server` and `## Health Check Path` in `.adw/commands.md` if scenarios require a running dev server
4. Run any of the five orchestrators normally — the scenario test/fix loop is applied automatically

## Configuration

| Setting | Location | Purpose |
|---|---|---|
| `MAX_TEST_RETRY_ATTEMPTS` | `adws/core/config.ts` (env var) | Maximum scenario test/fix iterations before continuing |
| `## Run Scenarios by Tag` | `.adw/commands.md` in target repo | Cucumber tag selector; `N/A` disables scenario testing |
| `## Start Dev Server` | `.adw/commands.md` in target repo | Command to start dev server before E2E scenario runs |
| `## Health Check Path` | `.adw/commands.md` in target repo | HTTP path for dev server health probe |

## Notes

- **Graceful skip** — when a target repo has no scenarios configured (`scenariosMd` empty or `runScenariosByTag: 'N/A'`), `scenarioTestPhase` returns immediately with a passing result and no retry iterations occur, so orchestrators running against non-BDD repos have zero overhead
- **Non-hard-fail on exhaustion** — if all `MAX_TEST_RETRY_ATTEMPTS` are consumed without scenarios passing, the workflow continues to the next phase (review, diff gate, PR), matching `adwSdlc` behaviour
- **`adwPlanBuildTest` has no scenario generation** — it relies on scenarios already present in the repo; no parallel `executeScenarioPhase` is added
- The `executePRReviewTestPhase` function in `prReviewCompletion.ts` is dead code for the orchestrator but is kept as an export; removal is deferred to a future cleanup slice
