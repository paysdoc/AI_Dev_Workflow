# JSONL Schema Probe + CI Conformance Check

**ADW ID:** `02r4w9-jsonl-schema-probe-c`
**Date:** 2026-03-24
**Specification:** `specs/issue-280-adw-02r4w9-jsonl-schema-probe-c-sdlc_planner-jsonl-schema-probe-ci-check.md`

## Overview

Introduces an automated three-layer defense to keep ADW's JSONL fixture files in sync with the real Claude CLI output schema. A schema probe captures the live envelope structure cheaply (~100 tokens), a conformance checker validates fixtures against it (CI entry point), and a fixture updater rewrites envelope fields while preserving hand-maintained payload content.

## What Was Built

- **Schema probe** (`adws/jsonl/schemaProbe.ts`) — spawns Claude CLI with `--print "say hello"`, captures JSONL, extracts per-message-type field schemas, writes `schema.json`
- **Canonical schema file** (`adws/jsonl/schema.json`) — committed to the repo; covers `assistant`, `result`, and `system` message types with required/optional field annotations
- **Fixture files** (`adws/jsonl/fixtures/`) — four canned `.jsonl` files: `assistant-text`, `assistant-tool-use`, `result-success`, `result-error`
- **Fixtures README** (`adws/jsonl/fixtures/README.md`) — documents the envelope/payload split convention
- **Conformance checker** (`adws/jsonl/conformanceCheck.ts`) — validates fixtures against the probed schema and through `parseJsonlOutput()` + `AnthropicTokenUsageExtractor`; CI entry point
- **Fixture updater** (`adws/jsonl/fixtureUpdater.ts`) — rewrites envelope fields in fixture files while preserving payload content (text, cost values, tool inputs)
- **Shared types** (`adws/jsonl/types.ts`) — `SchemaField`, `EnvelopeSchema`, `ConformanceResult`, `UpdateResult`
- **Barrel** (`adws/jsonl/index.ts`) — exports all public types and functions
- **npm scripts** — `jsonl:probe`, `jsonl:check`, `jsonl:update` wired into `package.json`

## Technical Implementation

### Files Modified

- `adws/jsonl/schemaProbe.ts`: New — spawns Claude CLI, extracts field schema recursively, writes `schema.json`
- `adws/jsonl/schema.json`: New — canonical envelope schema (committed; regenerated by `jsonl:probe`)
- `adws/jsonl/conformanceCheck.ts`: New — CI conformance checker with parse, schema, parser, and extractor checks
- `adws/jsonl/fixtureUpdater.ts`: New — programmatic fixture envelope updater with defined envelope/payload boundary
- `adws/jsonl/types.ts`: New — shared interfaces (`SchemaField`, `EnvelopeSchema`, `ConformanceResult`, `UpdateResult`)
- `adws/jsonl/index.ts`: New — barrel exports
- `adws/jsonl/fixtures/assistant-text.jsonl`: New — assistant message fixture with text content
- `adws/jsonl/fixtures/assistant-tool-use.jsonl`: New — assistant message fixture with tool_use content
- `adws/jsonl/fixtures/result-success.jsonl`: New — result message fixture with `modelUsage`
- `adws/jsonl/fixtures/result-error.jsonl`: New — minimal error result message fixture
- `adws/jsonl/fixtures/README.md`: New — envelope/payload split documentation
- `package.json`: Added `jsonl:probe`, `jsonl:check`, `jsonl:update` scripts

### Key Changes

- **Envelope/payload boundary** is explicitly defined per message type in `fixtureUpdater.ts` — envelope fields (structural metadata) are schema-controlled; payload fields (text, cost figures, tool inputs) are always preserved on update
- **Conformance checker runs four checks** per fixture: JSON parse validity, schema field coverage (missing required fields = fail; extra fields = informational), `parseJsonlOutput()` state-change verification, and `AnthropicTokenUsageExtractor.onChunk()` verification
- **Schema probe marks all observed fields as `required: true`** (one live example seen); the committed `schema.json` manually adjusts optional fields (e.g., `cache_read_input_tokens`, `durationMs`, `sessionId`)
- **Standalone executables**: all three scripts detect `isMain` via `process.argv[1]` comparison and run directly with `bunx tsx`, or import cleanly as modules
- **No new dependencies** — pure Node.js `fs`/`path`/`child_process` + existing ADW parsers

## How to Use

### Run the CI conformance check (standard use)

```bash
bun run jsonl:check
```

Exits `0` if all fixtures pass, `1` if any fail. Use this in CI. If it fails, the output shows exactly which fields are missing, which parser checks failed, and instructs you to run `jsonl:update`.

### Re-probe the schema against the live Claude CLI

```bash
bun run jsonl:probe
```

Spawns `claude --print --output-format stream-json "say hello"`, captures the JSONL envelope, and overwrites `adws/jsonl/schema.json`. Commit the updated `schema.json` to the repo afterwards.

### Programmatically update fixture envelopes after a schema change

```bash
bun run jsonl:update
```

Reads the current `schema.json`, updates envelope fields in all `.jsonl` fixture files, and reports what changed. Payload content (text, cost values, tool inputs) is preserved.

## Configuration

- **`CLAUDE_CODE_PATH`** (`.env`) — path to the Claude CLI binary. Used by `jsonl:probe` only. Falls back to `claude` on `PATH`.
- **`schema.json`** is committed to the repo so `jsonl:check` works in CI without a live Claude CLI.

## Testing

```bash
# Standard check — should exit 0 with all fixtures passing
bun run jsonl:check

# Full validation chain
bun run lint
bun run build
bunx tsc --noEmit -p adws/tsconfig.json
bun run jsonl:check
```

BDD scenarios covering the feature are in `features/jsonl_schema_probe_ci_check.feature`.

## Notes

- The `schema.json` is committed to the repo; `jsonl:probe` is only re-run when maintainers want to check for Claude CLI schema drift. CI does not re-probe.
- Unknown message types (present in schema but no fixture) produce an informational warning, not a failure.
- Malformed fixture files are reported individually; the checker continues processing remaining fixtures.
- The feature was unblocked from #275 (mock infrastructure) because the conformance check only reads `schema.json` and fixture files — it never spawns the Claude CLI.
