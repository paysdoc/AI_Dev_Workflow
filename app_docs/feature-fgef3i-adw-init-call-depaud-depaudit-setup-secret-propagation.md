# adw_init: depaudit Setup and Secret Propagation

**ADW ID:** fgef3i-adw-init-call-depaud
**Date:** 2026-04-17
**Specification:** specs/issue-439-adw-fgef3i-adw-init-call-depaud-sdlc_planner-adw-init-depaudit-setup-and-secret-propagation.md

## Overview

This feature hooks the `depaudit` supply-chain audit tool into ADW's target-repo bootstrap flow. After `adwInit.tsx` clones a target repo and copies target skills/commands, it now automatically invokes `depaudit setup` in the target repo worktree and propagates `SOCKET_API_TOKEN` and `SLACK_WEBHOOK_URL` from ADW's environment to the target repo's GitHub Actions secrets via `gh secret set`.

### Key Changes

- **`depaudit setup` invocation**: Called via `execWithRetry('depaudit setup', { cwd: config.worktreePath, maxAttempts: 2 })`; failures are caught and logged as warnings (never thrown), so a missing `depaudit` binary does not fail `adw_init`
- **Secret propagation**: `SOCKET_API_TOKEN` and `SLACK_WEBHOOK_URL` are piped via stdin to `gh secret set <NAME> --repo <owner>/<repo> --body -` — never inlined in argv to prevent leaking into process listings or shell history
- **`config.targetRepo` fallback**: When `config.targetRepo` is undefined (ADW running against its own repo), `getRepoInfo()` from `adws/github` resolves owner/repo from the current git remote
- **Warn-don't-fail contract**: Missing env vars and `gh` auth failures surface as entries in `skippedSecrets[]` and `warnings[]`; `success` is always `true` so the orchestrator continues to `commitChanges`
- **Phase ordering**: `executeDepauditSetup` runs before `commitChanges`, so files generated by `depaudit setup` (`.depaudit.yml`, `osv-scanner.toml`, `.github/workflows/depaudit.yml`) are included in the initialization commit

## How to Use

1. Install `depaudit` globally on the ADW host machine: `npm install -g depaudit`
2. Set `SOCKET_API_TOKEN` and `SLACK_WEBHOOK_URL` in your ADW `.env` file (see `.env.sample`)
3. Run `adw_init` as normal: `bunx tsx adws/adwInit.tsx <issueNumber>`
4. The `depaudit setup + secret propagation` phase runs automatically after skills/commands are copied
5. If either env var is missing, a warning is logged and the secret is listed in the init summary as "skipped" — the workflow continues normally

## Configuration

| Variable | Required | Description |
|---|---|---|
| `SOCKET_API_TOKEN` | Optional | Socket.dev API token — propagated to target repo GitHub Actions secrets via `gh secret set SOCKET_API_TOKEN` |
| `SLACK_WEBHOOK_URL` | Optional | Slack webhook for audit alerts — propagated to target repo GitHub Actions secrets via `gh secret set SLACK_WEBHOOK_URL` |

**Prerequisite**: `depaudit` must be installed globally (`npm install -g depaudit`). ADW does not install it during bootstrap.

**GitHub auth**: The `gh` auth context needs `actions:write` or `repo` scope on the target repo. Auth failures warn and continue.

## Notes

- **Security**: Secret values are always passed via `input` option (stdin) to `execWithRetry`, never interpolated into the command string. This prevents secrets from appearing in `ps` output, shell history, or ADW log files.
- **Non-retryable auth errors**: `execWithRetry` already treats `HTTP 401`, `gh auth login`, and `Bad credentials` as non-retryable (see `adws/core/utils.ts`). These fail fast and are caught as warnings.
- **Future extension**: The two `propagateSecret` calls iterate over `SECRET_NAMES = ['SOCKET_API_TOKEN', 'SLACK_WEBHOOK_URL'] as const`. To propagate additional secrets, add names to this array.
- **Out of scope**: ADW does not `npm install -g depaudit` during bootstrap — it is an operator prerequisite, consistent with how `gh` and `claude` CLIs are treated.
