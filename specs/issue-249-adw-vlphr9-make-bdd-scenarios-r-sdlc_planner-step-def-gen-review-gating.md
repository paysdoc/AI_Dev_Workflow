## Feature Description
This feature addresses a structural gap in the ADW workflow where BDD scenarios generated by the scenario agent have no step definitions, causing Cucumber to fail with undefined steps and blocking PR creation. The feature introduces:

1. A new `/generate_step_definitions` slash command and corresponding agent/phase that reads feature files, generates step definitions, and removes ungeneratable scenarios.
2. Restructured test phase to run only unit tests (BDD execution moves to the review phase).
3. Review phase decoupled from PR creation — runs on branch diff before PR exists.
4. Coding guidelines check added to the review command.
5. PR creation gated behind successful review in review-capable orchestrators.
6. Non-review orchestrators simplified by removing scenario writing, plan validation, and step def generation.

## User Story
As an ADW workflow operator
I want BDD scenarios to have step definitions generated automatically and the review phase to gate PR creation
So that Cucumber tests pass, review quality is enforced before PRs exist, and non-review workflows are leaner

## Problem Statement
The `/scenario_writer` command creates `.feature` files tagged `@adw-{issueNumber}` but never generates step definitions. When Cucumber runs during the test phase, all steps are undefined → non-zero exit → workflow fails → no PR created. Additionally, the current workflow creates PRs before review, meaning failed reviews still produce PRs, and non-review orchestrators wastefully run scenario writing and plan validation phases they don't need.

## Solution Statement
1. Create a new `/generate_step_definitions` slash command, agent wrapper (`stepDefAgent.ts`), and phase orchestrator (`stepDefPhase.ts`) that generates step definitions after the build phase, removes ungeneratable scenarios, and reports removals.
2. Strip BDD scenario execution from `testPhase.ts` — it becomes unit-tests-only.
3. Decouple `executeReviewPhase` from requiring a PR — it already works on branch diff via `git diff origin/<default>`.
4. Add a coding guidelines check to `review.md` (reads `.adw/coding_guidelines.md` with fallback to `guidelines/coding_guidelines.md`).
5. Reorder review-capable orchestrators: Plan → Scenarios → Validation → Build → Test (unit) → Step Def Gen → Review → [Document] → PR → [KPI].
6. Simplify non-review orchestrators: Plan → Build → Test (unit) → [Document] → PR.
7. Gate PR creation behind review pass — review failure posts error comment and exits.

## Step by Step Tasks

### Task 1: Create `/generate_step_definitions` slash command
- Create `.claude/commands/generate_step_definitions.md` with:
  - Variables: `$1` = issue number, `$2` = adwId
  - Instructions to read `.adw/scenarios.md` for the scenario directory path
  - Read all `@adw-{issueNumber}` tagged `.feature` files from the scenario directory
  - Read all existing step definition files in `features/step_definitions/` (or scenario directory's `step_definitions/` subfolder) to avoid duplicate patterns
  - Read implementation code from the worktree (source files relevant to the feature)
  - Generate step definitions that import Cucumber's `Given`, `When`, `Then` and match the step text patterns
  - Can create new files or modify existing step definition files
  - Identify ungeneratable scenarios (those requiring runtime infrastructure, mocked LLMs, external services, etc.)

## Testing Strategy

### Edge Cases
- Step def agent produces no output or fails → `stepDefPhase` catches error, logs warning, returns empty cost records (non-fatal)
- Step def agent removes all scenarios from a feature file → warning comment lists all removed scenarios; review still runs (may have only `@regression` scenarios)
- Review phase fails on first attempt → retry loop patches and re-runs up to `maxRetries` (existing behavior)
- Review phase exhausts retries → hard fail with `process.exit(1)`, error comment on issue, no PR created
- No `.adw/coding_guidelines.md` and no `guidelines/coding_guidelines.md` → coding guidelines check silently skipped
- Non-review orchestrators run without scenario/validation phases → clean execution path, no unused imports
- `testPhase` with unit tests disabled → logs "Unit tests disabled — skipping" and passes (unchanged behavior)
- `testPhase` with unit tests enabled → runs unit tests only, no BDD (BDD is now in review)
