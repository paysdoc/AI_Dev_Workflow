## Bug Description
When an ADW workflow crashes after creating a worktree and is restarted for the same issue, it fails to discover the existing worktree and instead creates a brand new one with a different branch name.

**Expected behavior:** On restart, `findWorktreeForIssue` should discover the existing worktree created by the first run (e.g., `bug-issue-29-clear-comment-triggers-adw`) and reuse it.

**Actual behavior:** `findWorktreeForIssue` fails to find the worktree because it searches using a regex pattern with prefix `bugfix-` (from `branchPrefixMap`), but the actual worktree directory name starts with `bug-` (generated by the `/generate_branch_name` Claude skill). A new worktree is then created with a different branch name (`bug-issue-29-fix-clear-comment-triggers-adw`).

## Problem Statement
`findWorktreeForIssue` in `adws/github/worktreeOperations.ts` constructs a regex pattern using `branchPrefixMap[issueType]` to match worktree directory names. For `/bug` issues, the prefix is `'bugfix'`, producing the pattern `^bugfix-issue-29-`. However, the `/generate_branch_name` Claude skill generates branch names like `bug-issue-29-...` (not `bugfix-issue-29-...`). The prefix mismatch causes the function to never find existing worktrees for bug issues.

This same mismatch exists for `/feature` issues when the Claude skill generates `feat-` instead of `feature-`, and potentially for `/pr_review` issues.

## Solution Statement
1. Add a `branchPrefixAliases` map in `adws/core/issueTypes.ts` that lists alternative prefixes the Claude skill may generate for each issue type.
2. Update `findWorktreeForIssue` to construct a regex that matches both the canonical `branchPrefixMap` prefix AND all known aliases.
3. Update the `/generate_branch_name` skill instructions and examples to align with `branchPrefixMap` prefixes (changing `bug-` to `bugfix-` and `feat-` to `feature-`), reducing future mismatches.
4. Add tests covering the alias matching behavior.

## Steps to Reproduce
1. Start an ADW workflow for a bug issue (e.g., issue #29): `npx tsx adws/adwPlanBuildTest.tsx 29`
2. The `/generate_branch_name` skill generates a branch like `bug-issue-29-clear-comment-triggers-adw`
3. A worktree is created at `.worktrees/bug-issue-29-clear-comment-triggers-adw`
4. Kill the process (simulate crash)
5. Restart the workflow for the same issue: `npx tsx adws/adwPlanBuildTest.tsx 29`
6. `findWorktreeForIssue('/bug', 29)` searches with pattern `^bugfix-issue-29-` — does NOT match `bug-issue-29-...`
7. If `recoveryState.branchName` is also not available, a new branch name is generated
8. A second, duplicate worktree is created

## Root Cause Analysis
There are **two independent branch naming systems** that use different prefix conventions:

1. **`generateBranchName()` in `gitOperations.ts`** uses `branchPrefixMap`:
   - `/bug` → `bugfix/issue-N-slug`
   - `/feature` → `feature/issue-N-slug`

2. **`/generate_branch_name` Claude skill** (`.claude/commands/generate_branch_name.md`) uses its own conventions:
   - `/bug` → `bug-issue-N-concise-name`
   - `/feature` → `feat-issue-N-concise-name` (sometimes `feature-`)

The actual branch names in production are generated by the Claude skill (called via `runGenerateBranchNameAgent`), not by `generateBranchName()`. But `findWorktreeForIssue` uses prefixes from `branchPrefixMap`, which assumes the `generateBranchName()` format.

The mismatch specifically:
| Issue Type | `branchPrefixMap` prefix | Skill-generated prefix | Match? |
|---|---|---|---|
| `/bug` | `bugfix` | `bug` | NO |
| `/feature` | `feature` | `feat` or `feature` | SOMETIMES |
| `/chore` | `chore` | `chore` | YES |
| `/pr_review` | `review` | varies | UNKNOWN |

Note: `removeWorktreesForIssue` in `worktreeCleanup.ts` correctly uses the pattern `-issue-${issueNumber}-` (no prefix constraint), which is why cleanup works but discovery doesn't.
