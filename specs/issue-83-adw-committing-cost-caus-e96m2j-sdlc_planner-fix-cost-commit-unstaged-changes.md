## Bug Description
When an issue is closed or a PR is merged/closed, the cost CSV commit flow calls `pullLatestCostBranch()` or `commitAndPushCostFiles()` which internally run `git pull --rebase origin "<branch>"`. This command fails when the working directory has unstaged or uncommitted changes, producing the error:

```
error: cannot pull with rebase: You have unstaged changes.
error: additionally, your index contains uncommitted changes.
error: please commit or stash them.
```

The expected behavior is that pulling latest changes before cost operations should succeed regardless of the working directory state, since cost operations only care about specific CSV files.

## Problem Statement
`git pull --rebase` requires a clean working directory. When cost CSV files are generated by the workflow (via `writeIssueCostCsv()` / `rebuildProjectCostCsv()`), or other concurrent operations leave modified files, the pull fails. This affects two locations:

1. `pullLatestCostBranch()` in `gitOperations.ts` (line 307) — called before cost revert in `trigger_webhook.ts` and before cost operations in `webhookHandlers.ts`
2. `commitAndPushCostFiles()` in `gitOperations.ts` (line 384) — called after committing cost files but before pushing; only cost-specific files are staged so other modified files remain unstaged

## Solution Statement
Add the `--autostash` flag to both `git pull --rebase` commands. The `--autostash` flag (available since git 2.9) automatically runs `git stash` before the rebase and `git stash pop` after, handling the dirty working directory transparently. This is a minimal, surgical change that directly addresses the root cause without altering the overall flow.

## Steps to Reproduce
1. Run an ADW workflow that generates cost CSV files (e.g., `npx tsx adws/adwPlanBuild.tsx 80`)
2. Close the associated issue (via GitHub UI or webhook)
3. The issue close handler in `trigger_webhook.ts` calls `pullLatestCostBranch()` while cost CSVs or other files are uncommitted in the ADW repo
4. `git pull --rebase origin "main"` fails with "You have unstaged changes"

## Root Cause Analysis
Issue #76 added `git pull --rebase` in two places to ensure cost commits are rebased on top of the latest remote state before pushing:

1. `pullLatestCostBranch()` (line 307 of `gitOperations.ts`) — a standalone function called by webhook handlers before cost operations
2. Inside `commitAndPushCostFiles()` (line 384 of `gitOperations.ts`) — called after committing cost files but before pushing

Both use plain `git pull --rebase origin "<branch>"` which requires a completely clean working directory (no staged or unstaged changes). However, when these run:
- Cost CSV files may have been written to disk but not yet staged/committed
- Other workflow operations may have left modified files in the working directory
- The `commitAndPushCostFiles()` function only stages specific cost files (not `git add -A`), so other modified files remain unstaged after the commit

The `--autostash` flag solves this by automatically stashing and restoring uncommitted changes around the rebase operation.
