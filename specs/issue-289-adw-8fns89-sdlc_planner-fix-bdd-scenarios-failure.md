## Bug Description
During the ADW workflow run for issue #278 (ADW ID `r4f0gi-application-type-con`), the BDD scenarios phase failed immediately after the unit tests were skipped. The error log shows:

```
📋 Unit tests disabled — skipping
📋 Phase: BDD Scenarios
❌ BDD scenarios failed. No PR was created.
```

The BDD scenarios failed within ~3 seconds, indicating a startup failure (missing step definitions) rather than a test assertion failure. The workflow exited without creating a PR.

**Expected behavior:** BDD scenarios should only run after step definitions have been generated (in the review phase), ensuring Cucumber can resolve all step patterns.

**Actual behavior:** BDD scenarios ran directly in the test phase via `runScenariosByTag` before step definitions existed, causing Cucumber to fail with undefined steps.

## Problem Statement
The test phase (`testPhase.ts`) was running BDD scenarios directly before step definitions were generated. This is a structural ordering issue: the scenario writer creates `.feature` files, but the corresponding step definition `.ts` files are generated by the step def agent, which runs AFTER the test phase. Running Cucumber with undefined steps causes an immediate non-zero exit code, which the test phase treated as a hard failure, aborting the workflow.

Additionally, the current `testPhase.ts` has a misleading log message: it unconditionally logs "Unit tests passed!" even when unit tests are disabled and no tests actually ran.

## Solution Statement
1. **Verify the primary fix is in place**: Confirm that BDD scenario execution has been fully removed from `testPhase.ts` and now runs exclusively in the review phase (via `regressionScenarioProof.ts` in `reviewRetry.ts`), where step definitions are guaranteed to exist.
2. **Fix misleading log**: Change the unconditional "Unit tests passed!" log message in `testPhase.ts` to only log when tests actually ran, and use a neutral message when tests are disabled.
3. **Add defensive pre-flight check**: Before running BDD scenarios (in the review phase), verify that at least one step definition file exists in the target repo's `features/step_definitions/` directory. If no step definitions exist, log a clear warning and skip the BDD proof gracefully instead of failing.
4. **Add workflow version logging**: Log the git commit hash at orchestrator startup so that stale-code issues are immediately diagnosable from the error log.

## Steps to Reproduce
1. Create a GitHub issue on a target repo that has `.adw/commands.md` with `## Run Scenarios by Tag` configured
2. Run the ADW SDLC orchestrator for that issue
3. The scenario writer creates `.feature` files tagged `@adw-{issueNumber}`
4. The build phase commits the implementation
5. The test phase runs — if it still contains BDD code, it calls `runScenariosByTag` which spawns `cucumber-js --tags "@adw-{issueNumber}"`
6. Cucumber fails because step definitions haven't been generated yet
7. The workflow aborts with "BDD scenarios failed. No PR was created."

## Root Cause Analysis
The root cause is a **phase ordering issue**: BDD scenarios were executed in the test phase (step 5 of the workflow) before the step definition generation phase (step 6). The step def agent generates the TypeScript step definition files that Cucumber needs to match `Given/When/Then` steps. Without these files, Cucumber reports undefined steps and exits with a non-zero code.

The fix was applied in commit `8507da4` (PR #252, merged 2026-03-21) which:
- Removed BDD scenario execution from `testPhase.ts`
- Added `stepDefPhase.ts` to generate step definitions
- Moved BDD scenario execution to the review phase via `regressionScenarioProof.ts`

The error on 2026-03-25 occurred because the workflow was running from a code checkout that hadn't incorporated this fix (the error messages match the pre-fix version of `testPhase.ts`).

The remaining issues in the current code are:
1. `testPhase.ts` line 97 unconditionally logs "Unit tests passed!" even when tests are disabled
2. No pre-flight check exists to verify step definitions are present before running BDD scenarios
3. No version/commit logging at orchestrator startup to diagnose stale-code issues
